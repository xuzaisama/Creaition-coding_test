基于 [改进版.md](/Users/xuzai/Desktop/Creaition-coding_test/改进版.md) 的要求，这里给出一份适合面试作业提交的项目解决方案。

- **完成范围：核心 CRUD + 筛选排序分页 + 任务依赖 + 基础缓存 + 架构说明**

## 2. 项目目标

构建一个任务管理系统后端，支持用户创建、查询、更新、删除任务，并提供如下增强能力：

- 按状态、优先级、标签筛选任务
- 按创建时间、优先级、状态排序
- 分页查询
- 管理任务之间的依赖关系
- 在依赖未完成时，禁止将任务标记为完成
- 提供依赖树查询接口
- 为高频读取接口增加缓存
- 保持良好的工程结构、错误处理、输入校验和文档质量

## 3. 技术选型

### 推荐技术栈

- **语言：** Python 3.11
- **后端框架：** FastAPI
- **ORM：** SQLAlchemy 2.x
- **数据校验：** Pydantic v2
- **数据库：** SQLite（挑战版） / PostgreSQL（进阶版）
- **缓存：** 内存 TTL Cache（挑战版） / Redis（进阶版）
- **测试：** pytest
- **容器化：** Docker + Docker Compose

### 选型理由

- `FastAPI` 上手快，自动生成 OpenAPI 文档，非常适合面试项目。
- `SQLAlchemy` 能让模型层、查询层更清晰，也便于未来切换数据库。
- `Pydantic` 非常适合做请求参数校验和响应结构约束。
- `SQLite` 适合在有限时间内快速交付可运行版本。
- 若想体现更强工程能力，可在 README 中说明生产环境建议切换到 `PostgreSQL + Redis`。

### 建议的取舍

- **提交版本优先可运行：** FastAPI + SQLite + 本地缓存
- **设计层面体现扩展性：** 代码结构按 PostgreSQL/Redis 可替换方式设计

## 4. 系统架构

采用典型分层架构：

1. **API 层**
负责接收 HTTP 请求、参数校验、返回标准响应。

2. **Service 层**
承载业务逻辑，例如：
- 任务状态流转
- 依赖校验
- 缓存失效
- 搜索 / 筛选 / 排序策略

3. **Repository / Data Access 层**
负责数据库读写，隔离 ORM 细节。

4. **Model 层**
定义数据库实体与关联关系。

5. **Schema 层**
定义请求体、响应体、分页结果等结构。

### 架构图

```text
Client
  |
  v
FastAPI Router
  |
  v
Task Service
  |
  +--> Task Repository --> SQLite / PostgreSQL
  |
  +--> Cache Layer -----> In-memory TTL / Redis
```

## 5. 核心功能设计

### 5.1 任务 CRUD

支持以下基础能力：

- 创建任务
- 查询单个任务
- 查询任务列表
- 更新任务
- 删除任务

### 5.2 筛选、排序、分页

`GET /tasks` 支持：

- `status`
- `priority`
- `tags`
- `sort_by`
- `order`
- `page`
- `page_size`

### 5.3 任务依赖关系

支持任务依赖其他任务，例如：

- 任务 B 依赖任务 A
- 若 A 未完成，则 B 不允许变更为 `completed`

同时增加以下能力：

- 添加依赖关系
- 删除依赖关系
- 查询某个任务的依赖树
- 防止循环依赖

### 5.4 搜索功能

作为建议加分项，可增加：

- 根据 `title` / `description` 关键词搜索

挑战版中可使用 SQL `LIKE`；
进阶版中可升级为 PostgreSQL Full-Text Search。

## 6. 数据模型设计

### 6.1 tasks 表

字段建议如下：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| id | UUID / Integer | 主键 |
| title | String | 标题，必填 |
| description | Text | 描述，可选 |
| status | Enum | `pending / in_progress / completed` |
| priority | Enum | `low / medium / high` |
| created_at | DateTime | 创建时间 |
| updated_at | DateTime | 更新时间 |

### 6.2 task_tags 表

由于 SQLite 对数组支持一般，建议将标签拆表：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| id | Integer | 主键 |
| task_id | FK | 关联任务 |
| tag | String | 标签值 |

### 6.3 task_dependencies 表

用于表示任务依赖关系：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| id | Integer | 主键 |
| task_id | FK | 当前任务 |
| depends_on_task_id | FK | 被依赖任务 |

### 关键约束

- `title` 非空
- `status`、`priority` 必须在枚举范围内
- `task_id != depends_on_task_id`
- 同一依赖关系不能重复创建
- 不允许形成环

## 7. API 设计

### 基础接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/tasks` | 创建任务 |

## 8. 关键业务规则

### 8.1 状态流转规则

- 新建任务默认状态为 `pending`
- 可从 `pending` 变更为 `in_progress`
- 可从 `in_progress` 变更为 `completed`
- 若任务存在未完成依赖，则禁止设置为 `completed`

### 8.2 删除规则

建议采用以下策略之一：

- **简单策略：** 若任务被其他任务依赖，则禁止删除
- **备选策略：** 删除任务时同步删除相关依赖关系

面试作业中更推荐第一种，因为规则清晰，也更安全。

### 8.3 依赖关系规则

- 不能依赖自己
- 不能创建重复依赖
- 不能创建循环依赖

例如：

- A 依赖 B
- B 依赖 C
- 则 C 不能再依赖 A

## 9. 性能优化方案

为了满足题目中的性能要求，可实现以下优化：

### 9.1 索引优化

为以下字段建立索引：

- `status`
- `priority`
- `created_at`
- `title`

关联表中建立：

- `task_id`
- `depends_on_task_id`
- `tag`

### 9.2 查询优化

- 列表接口采用分页，避免全量返回
- 标签查询采用联表或子查询
- 依赖树查询可在应用层做 BFS/DFS

### 9.3 缓存策略

建议缓存以下内容：

- 任务详情 `GET /tasks/{id}`
- 高频列表查询 `GET /tasks`

缓存策略：

- Key 示例：`task:{id}`、`task_list:{query_hash}`
- TTL：30-60 秒
- 当任务被创建、更新、删除时，主动清理对应缓存

### 9.4 响应时间目标

在本地开发环境下，基础 CRUD 接口目标：

- 单任务查询：< 50ms
- 创建 / 更新：< 100ms
- 列表查询：< 100ms

## 10. 面向 10 万任务的扩展设计

虽然挑战项目未必真的实现大规模部署，但需要在方案中体现可扩展思路。

### 扩展方向

- 数据库从 SQLite 升级为 PostgreSQL
- 缓存层替换为 Redis
- 引入异步任务处理复杂统计或全文索引
- 使用分页和索引降低列表查询压力
- API 层可横向扩容

### 可进一步演进的架构

- `App Server` 多实例部署
- `PostgreSQL` 主从或托管数据库
- `Redis` 做热点数据缓存
- `Nginx` 做反向代理和限流

## 12. 异常处理方案

建议统一错误返回格式，例如：

```json
{
  "error": "DEPENDENCY_NOT_COMPLETED",
  "message": "Cannot complete task before dependencies are completed"
}
```

常见错误场景：

- 参数校验失败：`400`
- 资源不存在：`404`
- 业务规则冲突：`409`
- 服务内部错误：`500`
