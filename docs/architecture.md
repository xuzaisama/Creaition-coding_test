# 架构说明

## 目标

本项目实现一个可运行的任务管理系统，重点覆盖后端方向要求，并提供一个轻量前端控制台用于演示完整流程。

当前架构关注以下几点：

- 在较短挑战时间内保持代码结构清晰
- 支持任务 CRUD、筛选、排序、分页、搜索和依赖关系
- 使用 SQLite 提供本地持久化能力
- 通过缓存优化高频读取接口
- 让前端页面直接消费同一套 API，避免额外的中间层

## 系统概览

系统由两部分组成：

1. FastAPI 后端
2. 原生 HTML / CSS / JavaScript 前端

后端同时承担 API 服务和静态资源分发职责：

- `/api/*` 提供任务相关接口
- `/health` 提供健康检查
- `/` 返回前端 Dashboard
- `/assets/*` 提供前端静态资源

## 分层设计

项目采用 `Router -> Service -> Repository -> Database` 的分层结构。

### Router 层

位置：`app/api/task_routes.py`

职责：

- 定义 HTTP 接口
- 处理查询参数和请求体
- 组织响应结构
- 接入缓存命中标识
- 在写操作后触发读缓存失效

### Service 层

位置：`app/services/task_service.py`

职责：

- 承载业务规则
- 校验依赖关系相关约束
- 处理任务完成、删除、依赖添加等业务限制
- 构建依赖树

这里放的是“业务为什么能做/不能做”的逻辑，而不是数据库细节。

### Repository 层

位置：`app/repositories/task_repository.py`

职责：

- 负责 SQLAlchemy 查询和持久化
- 处理筛选、排序、分页
- 封装依赖关系查询
- 封装标签替换、搜索和计数逻辑

这里放的是“怎么查、怎么写数据库”的逻辑。

### Model / Schema 层

位置：

- `app/models/task.py`
- `app/schemas/task.py`

职责：

- Model 定义数据库表结构和关系
- Schema 定义请求体、响应体和参数校验规则

### Core 层

位置：

- `app/core/config.py`
- `app/core/database.py`
- `app/core/cache.py`

职责：

- 环境配置
- 数据库连接和初始化
- 进程内 TTL 缓存

## 数据模型

当前主要包含三张表：

### `tasks`

存储任务主体信息：

- `id`
- `title`
- `description`
- `status`
- `priority`
- `created_at`
- `updated_at`

### `task_tags`

存储任务标签，一条任务可关联多个标签。

这样设计比把标签塞进单字段更利于：

- 标签筛选
- 去重约束
- 后续统计与扩展

### `task_dependencies`

存储任务之间的依赖关系。

这样设计便于支持：

- 一个任务依赖多个前置任务
- 查询直接依赖
- 构建依赖树
- 检测循环依赖

## 核心请求流

以“创建任务”为例：

1. 前端或调用方向 `POST /api/tasks` 发送请求
2. Router 接收请求并完成参数解析
3. Schema 负责字段校验
4. Service 调用 Repository 写入数据库
5. Repository 提交事务并返回最新任务对象
6. Router 清理相关读缓存
7. 前端重新拉取任务列表和汇总数据

以“查询任务列表”为例：

1. Router 根据查询参数生成缓存键
2. 如果命中缓存，直接返回缓存结果并标记 `X-Cache: HIT`
3. 如果未命中，则调用 Service / Repository 查询数据库
4. 将结果写入缓存并返回 `X-Cache: MISS`

## 缓存策略

当前使用进程内 TTL 缓存，适用于单机开发和挑战演示场景。

缓存的接口包括：

- 任务列表
- 任务搜索
- 任务详情
- 依赖树查询

缓存策略特点：

- 读请求优先走缓存
- 写请求后统一按前缀失效
- 实现简单，便于解释和调试

当前方案的限制：

- 仅在单进程内有效
- 不适合多实例部署
- 不适合作为分布式缓存方案

如果后续升级，可替换为 Redis。

## 前端架构

前端位于 `app/static/`，使用原生 JavaScript 管理状态与交互。

前端职责：

- 渲染任务列表、详情和依赖信息
- 提交创建、编辑、删除、状态流转请求
- 提供搜索、筛选、标签选择和分页
- 展示服务状态、任务统计和缓存命中信息

前端与后端的关系保持简单直接：

- 所有数据都来自同一套后端 API
- 页面刷新后重新读取数据库中的真实数据
- 汇总统计通过读取任务集合重新计算，而不是维护独立统计表

## 错误处理

后端在 `app/main.py` 中注册了统一异常处理器。

这样做的好处：

- API 响应格式一致
- 前端更容易统一显示错误信息
- 业务冲突、校验失败、资源不存在等情况更容易区分

典型错误类型包括：

- `validation_error`
- `not_found`
- `dependency_not_found`
- `conflict`
- `internal_server_error`

## 部署与运行

当前支持三种交付方式：

- 本地直接运行 `uvicorn`
- Docker 单容器运行
- Docker Compose 启动

项目还包含基础 GitHub Actions CI，用于在推送后自动运行测试。

## 面向扩展的考虑

如果系统需要扩展到更多任务量或更复杂的使用场景，可以沿以下方向演进：

### 数据库层

- 从 SQLite 切换到 PostgreSQL
- 引入数据库迁移工具，例如 Alembic
- 为高频筛选条件增加更明确的索引策略

### 缓存层

- 将进程内缓存替换为 Redis
- 针对列表查询和详情查询制定更细粒度的失效策略

### 搜索能力

- 从 `LIKE / ILIKE` 升级到全文检索
- 在更复杂场景下增加语义搜索

### 系统能力

- 增加用户系统、认证与授权
- 增加限流和审计日志
- 拆分前后端部署
- 增加更丰富的监控和可观测性

## 当前取舍

本项目优先选择“结构清晰、可运行、可验证”的实现方式，因此有以下明确取舍：

- 使用 SQLite 以降低启动和演示成本
- 使用原生前端以减少框架复杂度
- 使用进程内缓存以换取实现速度
- 以 API 集成测试为主，而不是完整覆盖所有细粒度单元测试

这套架构适合作为 coding challenge 的交付基础，也为后续继续演进留下了清晰路径。
